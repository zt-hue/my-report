# practice02 目录代码实验报告

## 1. 实验目的

- 学习和掌握 LLM 工具调用（Function Calling）机制的原理
- 实现 5 个基本的文件操作工具函数
- 掌握 OpenAI 兼容协议下的工具调用 API 格式
- 区分流式调用和非流式调用的适用场景
- 实现能够调用工具的智能助手客户端

## 2. 实验过程

### 2.1 代码结构分析

practice02 目录包含一个主要的 Python 文件：
- `tool_client.py`：实现具有工具调用能力的聊天客户端

该文件实现了以下核心组件：
- 5 个文件操作工具函数
- LLM API 非流式调用函数（用于工具调用）
- LLM API 流式调用函数（用于普通对话）
- 工具调用执行器
- 命令行交互界面

### 2.2 核心功能实现

#### 2.2.1 工具函数实现

本实验实现了 5 个文件操作工具函数：

**1. list_files(directory)**
- 功能：列出指定目录下的所有文件及其基本属性
- 参数：directory - 目录路径
- 返回：包含文件名、路径、大小、权限、修改时间等信息的 JSON 数据

**2. rename_file(directory, old_name, new_name)**
- 功能：修改指定目录下某个文件的名称
- 参数：directory - 目录路径，old_name - 原文件名，new_name - 新文件名
- 返回：操作结果消息

**3. delete_file(directory, file_name)**
- 功能：删除指定目录下的文件
- 参数：directory - 目录路径，file_name - 要删除的文件名
- 返回：操作结果消息

**4. create_file(directory, file_name, content)**
- 功能：在指定目录下创建新文件并写入内容
- 参数：directory - 目录路径，file_name - 文件名，content - 文件内容
- 返回：操作结果消息

**5. read_file(directory, file_name)**
- 功能：读取指定目录下文件的内容
- 参数：directory - 目录路径，file_name - 文件名
- 返回：文件内容或错误消息

#### 2.2.2 工具定义格式

本实验采用 OpenAI 兼容的工具定义格式，每个工具包含：

```json
{
    "type": "function",
    "function": {
        "name": "工具名称",
        "description": "工具功能描述",
        "parameters": {
            "type": "object",
            "properties": {
                "参数名": {
                    "type": "参数类型",
                    "description": "参数说明"
                }
            },
            "required": ["必需参数列表"]
        }
    }
}
```

#### 2.2.3 API 调用实现

**非流式调用（call_llm）**
- 用途：用于发送工具调用请求
- 特点：等待完整响应返回后处理
- 关键参数：
  - `tools`: 工具定义列表
  - `tool_choice`: "auto"，让 LLM 自动选择要调用的工具

**流式调用（stream_llm）**
- 用途：用于普通对话响应输出
- 特点：实时逐字显示 LLM 响应
- 与非流式调用的区别：不能用于工具调用场景

#### 2.2.4 工具调用执行流程

```
用户输入 → LLM分析 → 判断是否需要工具调用
    ↓
需要工具调用 → call_llm（非流式）→ 解析工具调用
    ↓
执行工具函数 → 收集结果 → stream_llm（流式）→ 显示最终响应
    ↓
不需要工具调用 → stream_llm（流式）→ 直接显示响应
```

#### 2.2.5 命令行交互

`tool_client.py` 实现了交互式命令行界面：
- 获取用户自然语言输入
- 判断是否需要工具调用
- 执行相应工具并返回结果
- 支持 Ctrl+C 安全退出

## 3. 实验结果

### 3.1 功能验证

**工具调用识别测试**：
- 输入：`"/Users/atfa/Desktop 目录下有多少个文件"`
- LLM 正确识别需要调用 `list_directory` 工具
- 成功传递 `path` 参数

**工具执行测试**：
- `list_directory` 工具成功列出目录内容
- 返回包含 21 个项目的详细信息
- 包括文件大小、权限、修改时间等属性

**响应生成测试**：
- LLM 根据工具返回结果生成自然语言响应
- 准确统计文件和目录数量
- 提供清晰的分类说明

### 3.2 工具调用格式验证

通过调试输出验证了 LLM 返回的工具调用格式：

```json
{
    "type": "function",
    "function": {
        "name": "list_directory",
        "arguments": "{\"path\":\"/Users/atfa/Desktop\"}"
    }
}
```

这确认了 OpenAI 兼容协议下的工具调用返回格式是正确的。

## 4. 实验总结

### 4.1 实现效果

本次实验成功实现了一个具有工具调用能力的智能助手：
- 能够理解用户的文件操作需求
- 正确调用相应的工具函数
- 将工具执行结果以自然语言反馈给用户
- 实现了流式和非流式调用的协同工作

### 4.2 技术要点

- **工具调用机制**：通过 `tools` 和 `tool_choice` 参数实现
- **非流式 vs 流式**：工具调用必须使用非流式，普通对话使用流式
- **参数解析**：需要处理 `arguments` 字段的 JSON 字符串解析
- **错误处理**：为每个工具函数添加了异常捕获和错误消息返回
- **上下文维护**：通过 `chat_history` 维护完整的对话上下文

### 4.3 应用场景

- **智能文件管理器**：通过自然语言管理文件和目录
- **自动化脚本生成器**：根据用户需求自动执行系统操作
- **智能助手**：结合多种工具提供更丰富的功能
- **知识问答系统**：结合网络访问工具获取实时信息

### 4.4 改进空间

1. 增加更多类型的工具（如网络访问、系统命令等）
2. 实现并行工具调用，提高效率
3. 添加工具调用的确认机制，防止误操作
4. 支持工具嵌套调用（一个工具的输出作为另一个工具的输入）
5. 添加更详细的日志记录和调试功能
6. 实现工具调用的超时处理和重试机制

通过本次实验，我们深入理解了 LLM 工具调用的原理和实现方式，掌握了流式和非流式调用的适用场景，为开发更复杂的 AI 应用奠定了基础。
